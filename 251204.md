### 결과 

결론부터 말하면,

**“친구 시드와 일기 시드 처리가 동일 메서드 내부에 섞여 있는 현재 구조는 SRP(Single Responsibility Principle, 단일 책임 원칙)를 명확히 위반하는 구조”**가 맞아!

왜 SRP 위반인지 아주 명확하게 설명해줄게.

## SeedInitializer.seed() 내부 로직 분석
```java
현재 코드:

public static void seed(TaskViewModel taskVM,
                        ScheduleViewModel schedVM,
                        FriendViewModel friendVM,
                        CalendarViewModel calVM) {

    friendVM.add("민수", "uid_123");
    friendVM.add("지은", "uid_456");

    calVM.addDiary(LocalDate.now(), "중간고사 대비", null);
}
```

이 seed() 메서드가 하는 일은?

- 친구 데이터 시드

- 오늘 일기 데이터 시드

즉, 최소 두 가지 책임(행동)을 수행하고 있음

## SRP 정의 간단 이해

### SRP(단일 책임 원칙)

“하나의 클래스 또는 메서드는 오직 하나의 책임만 가져야 한다.”

여기서 **책임(Responsibility)**은

비즈니스 규칙, 변경 이유, 기능 관심사 등을 의미함.

## 이 코드가 SRP를 위반하는 이유

###  ① 변경 이유가 최소 2개 이상 존재

“친구 시드 데이터 로직을 바꾸게 된다면?”

“일기 시드 데이터 형태가 바뀌게 된다면?”

둘 중 하나라도 변경되면 seed() 메서드를 수정해야 함

SRP 위반 대표 사례

### ② 두 개의 독립된 관심사를 한 곳에서 처리하고 있음
관심사	설명
 
Friend Seed	VM 호출, ID/Name/UID 데이터 제공
 
Diary Seed	LocalDate.today, 일기 내용 입력

두 로직은 서로 관련성이 없음
-> 자연스럽게 서로 다른 책임임

따라서 하나의 seed() 메서드에서 같이 처리하는 것은 부적절함

### ③ 확장 시 유지보수가 어려워짐

만약 이후 시드가 더 늘어난다면?

```java
예:

seedTasks();
seedGrades();
seedSchedules();
```

현재 구조에서는 전부 seed() 안에 넣어야 함
-> seed() 메서드가 계속 비대해짐

Fat Method(비대메서드) 발생

## SRP 위반 지표를 적용해보면?

SRP 판단 기준	현재 코드 상태
변경 이유가 한 가지인가?	X (친구 시드 변경, 일기 시드 변경)
기능 목적이 하나인가?	X (두 개 이상)
관심사가 명확하게 분리되어 있는가?	X (한 메서드에 섞임)

따라서 단일 책임 원칙 위반 확정

## “왜 SRP가 중요한가?”

### SRP를 지키면?


- 수정 영향 범위 최소화
- 유지보수 쉬움
- 테스트 단위 분리 가능
- 구조 명확
- 관심사별 변경 독립 가능

### SRP를 어기면?

- seed()가 점점 거대해짐
- 기능 추가할수록 코드 난잡
- 특정 seed 기능 수정 시 다른 seed가 문제 생길 가능성

## 리팩토링 방향 (정답 예시)

개선 버전
```java
public class SeedInitializer {
    public static void seed(
            TaskViewModel taskVM,
            ScheduleViewModel schedVM,
            FriendViewModel friendVM,
            CalendarViewModel calVM) {

        seedFriends(friendVM);
        seedDiary(calVM);
    }

    private static void seedFriends(FriendViewModel friendVM) { ... }

    private static void seedDiary(CalendarViewModel calVM) { ... }
}
```

SRP 관점에서 이점
| 장점       | 내용                                        |
|------------|---------------------------------------------|
| 책임 분리  | friend 시드와 diary 시드가 독립             |
| 변경 격리  | 친구 관련 로직만 바뀌면 `seedFriends()`만 수정 |
| 테스트 용이 | 각 시드를 따로 테스트 가능                  |
| 확장성     | 새로운 시드는 새 함수로 쉽게 추가           |
